# 렌더링 방식

# MPA

라우트마다 **새 HTML**을 받음(전통적 서버 렌더링)

# SSR

# SSG

빌드할 때 미리 HTML을 만들어 두고, 요청이 오면 그 **정적 파일**을 그대로 서빙”하는 방식이야. 첫 응답이 빠르고(CDN 캐시 활용), 서버 부하가 작다는 게 장점이야. 보통 **초기 HTML은 정적**으로 보내고, 브라우저가 이후에 JS로 **하이드레이션**해 상호작용을 붙여서 SPA처럼 동작해.

## 언제 쓰면 좋은가

- **자주 안 바뀌는 공개 페이지**: 블로그/문서/마케팅 랜딩/카탈로그 등
- **SEO·첫 화면 속도**가 중요한 곳: 서버 연산 없이 바로 HTML 응답
- **대규모 트래픽**: CDN에 캐시해 비용·응답시간을 줄이기 쉽다
- **개인화/실시간**에는 부적합: 로그인별 다른 화면은 SSR/CSR로

# SPA

페이지 전체를 다시 받지 않고, 한 번 받은 페이지 안에서 자바스크립트로 화면을 바꿔 가는 방식”**이야. 서버는 주로 **데이터(JSON)**만 주고, **라우팅·렌더링은 브라우저(클라이언트)**가 맡아.

## 핵심 개념

- **초기 로드 1번**: `/index.html`과 큰 JS 번들을 한 번 내려받고
    
    → 이후 이동은 **History API(pushState)**와 **클라이언트 라우터**가 처리.
    
- **클라이언트 렌더링(CSR)**: 화면 조립을 JS가 수행(React/Vue 등).
- **API 중심 통신**: 페이지가 아니라 **/api/**에서 JSON을 받아 상태만 갱신.

## 언제 잘 맞나

- **앱형 UI**: 대시보드, 이메일/채팅, 생산성 도구처럼 **상태가 복잡**하고 **자주 상호작용**하는 서비스.
- **페이지 전환 부드럽게**: 전체 새로고침 없이 뷰만 갈아끼우니 UX가 매끄러워.

# SSR+SPA(혼합)

**서버에서 초안 HTML** 제공 → 브라우저가 **하이드레이트** 후부터는 SPA처럼 동작

# ISR

*빌드 후에도* 정적으로 만든 페이지를 **선택적으로 다시 생성**(regenerate)해 최신화하는 Next.js 기능이야. 즉, SSG의 속도/비용 장점은 유지하면서도 **데이터가 바뀐 페이지만** 새로 만들어 캐시에 반영하지
