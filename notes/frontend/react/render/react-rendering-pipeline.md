# 렌더링 파이프라인

React에서 화면이 업데이트되는 과정은 흔히 **"렌더링(Rendering)"** 한 단어로 표현되곤 하지만, 내부적으로는 **Render 단계**와 **Commit 단계**라는 두 가지 명확한 과정을 거칩니다.

이 과정을 이해하기 쉽게 **음식점(레스토랑)**에 비유해서 설명해 드릴게요.

---

### 1. Trigger (주문하기)

렌더링 과정이 시작되려면 계기가 필요합니다. 보통 **컴포넌트의 상태(State)가 변하거나**, **부모로부터 새로운 Props를 받았을 때** 발생합니다. 레고 블록으로 치면 "이 부분 색깔을 바꿔줘!"라고 요청이 들어온 상태입니다.

### 2. Render Phase (주방에서 요리법 결정하기)

이 단계는 React가 **"화면에 무엇을 그려야 할지"** 결정하는 단계입니다. 실제 화면(DOM)을 건드리지 않고 내부적으로 계산만 수행합니다.

- **컴포넌트 호출:** React가 해당 컴포넌트 함수를 실행합니다.
- **Virtual DOM 생성:** 새로운 Virtual DOM 트리를 만듭니다.
- **Diffing (비교):** 이전 Virtual DOM과 새로 만든 트리를 비교하여 **바뀐 부분(차이점)**이 무엇인지 찾아냅니다.
- **특징:** 이 과정은 엔진 내부에서만 일어나기 때문에 매우 빠르며, 사용자 눈에는 아무 변화도 보이지 않습니다.

### 3. Commit Phase (음식 서빙 및 테이블 세팅)

React가 바뀐 내용을 찾아냈다면, 이제 **실제 화면(DOM)에 적용**하는 단계입니다.

- **DOM 조작:** Render 단계에서 찾아낸 '차이점'들만 실제 브라우저의 DOM에 반영(삽입, 수정, 삭제)합니다.
- **일괄 처리:** 성능을 위해 한꺼번에 업데이트합니다.
- **결과:** 이 단계가 끝나야 브라우저의 실제 요소들이 변경됩니다.

### 4. Browser Paint (손님이 음식 먹기)

엄밀히 말하면 React의 단계는 아니지만, Commit이 끝난 후 브라우저가 화면을 다시 그리는 과정입니다. 사용자는 이 시점에 비로소 변화된 화면을 보게 됩니다.
