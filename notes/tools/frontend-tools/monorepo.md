# MonoRepo

# 정의

여러 패키지·앱(라이브러리, 서비스, 디자인 시스템 등)을 **한 저장소(repo)** 안에서 함께 관리하는 방식입니다. (반대: **Polyrepo**—프로젝트별로 저장소를 따로 둠)

# 왜 쓰나 (장점)

- **코드 공유/재사용↑**: 공통 UI/유틸을 패키지로 뽑아 여러 앱이 함께 사용.
- **원자적 변경**: A, B, C 패키지에 걸친 대규모 리팩터링을 **한 PR**로 처리.
- **일관된 규칙**: ESLint/Prettier/TS 설정, CI 파이프라인을 중앙에서 통일.
- **버전/의존성 관리 효율**: workspace 도구가 루트에서 의존성 중복을 줄이고 설치 속도↑.
- **빌드/테스트 가속**: 변경 영향만 골라 실행(affected), 캐시/원격 캐시로 시간 절약.

# 단점/주의

- **의존성 지옥(호이스팅 이슈)**: 잘못하면 “우연히 통과”하는 트랜스티브 의존성에 의존.
- **경계 약화**: 레이어/패키지 경계가 모호해져서 순환참조나 내부 API 오남용 가능.
- **도구 난이도**: 빌드 그래프, 캐시, 릴리스(버저닝) 파이프라인 설계가 필요.
